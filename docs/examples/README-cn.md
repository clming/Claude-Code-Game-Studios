# 协作式会话示例

本目录包含真实、端到端的会话记录示例，用来展示 Game Studio Agent Architecture 在实际工作中的运行方式。每个示例都演示了**协作式工作流**：代理会先提问、给出选项，并等待用户批准，而不是自主生成内容后直接执行。

---

## 可用示例

### [会话：设计制作系统](session-design-crafting-system.md)
**类型：** 设计  
**代理：** `game-designer`  
**时长：** 约 45 分钟（12 轮）  
**复杂度：** 中等

**场景：**  
独立开发者需要设计一个制作系统，以支撑支柱 2“通过实验实现涌现式发现（Emergent Discovery Through Experimentation）”。代理会通过问答引导用户，给出 3 个带有博弈设计分析的方案，吸收用户修改意见，并在每一步都先征得同意，再逐步撰写 GDD。

**关键协作时刻：**
- 代理一开始先提出 5 个澄清问题
- 给出 3 个不同方案，并分析优缺点与 MDA 对齐方式
- 用户修改推荐方案后，代理立即吸收并调整
- 主动标出边界情况（“如果不是配方组合怎么办？”）
- 每个 GDD 小节都会先展示并等待批准，再继续下一节
- 创建文件前明确询问“我可以写入 [file] 吗？”

**你将学到：**
- 设计类代理如何询问目标、约束和参考资料
- 如何用游戏设计理论（MDA、SDT、Bartle）来呈现方案
- 如何按章节迭代草稿
- 何时委派给专家代理（`systems-designer`、`economy-designer`）

### [会话：实现战斗伤害计算](session-implement-combat-damage.md)
**类型：** 实现  
**代理：** `gameplay-programmer`  
**时长：** 约 30 分钟（10 轮）  
**复杂度：** 低到中等

**场景：**  
用户已经有完整的设计文档，希望实现伤害计算逻辑。代理会先阅读规格，识别出 7 个不明确或缺失之处，提出澄清问题，给出待批准的架构方案，在规则约束下完成实现，并主动编写测试。

**关键协作时刻：**
- 代理先阅读设计文档，再识别出 7 处规格歧义
- 在实现之前先给出架构方案和代码示例
- 用户要求更强的类型安全后，代理会重新细化并再次提案
- 规则检查捕获问题（如硬编码值），代理会透明地修正
- 依据“验证驱动开发”主动补上测试
- 代理会提供后续选项，而不是自行假设下一步

**你将学到：**
- 实现类代理在编码前如何澄清规格
- 如何先给出带代码示例的架构方案供批准
- 规则如何自动执行标准
- 如何处理规格缺口（先询问，不自行假设）
- 验证驱动开发（用测试证明它能工作）

### [会话：范围危机 - 战略决策](session-scope-crisis-decision.md)
**类型：** 战略决策  
**代理：** `creative-director`  
**时长：** 约 25 分钟（8 轮）  
**复杂度：** 高

**场景：**  
独立开发者面临危机：Alpha 里程碑还剩 2 周，制作系统需要 3 周，而投资人演示成败攸关。创意总监会先收集背景，搭建决策框架，给出 3 个带有真实权衡分析的战略选项，提出建议但把最终决定留给用户，然后用 ADR 和演示脚本记录决策结果。

**关键协作时刻：**
- 代理先阅读背景文档，再提出解决方案
- 通过 5 个问题理解决策约束
- 正确框定决策问题（赌注是什么、评估标准是什么）
- 给出 3 个选项，并附上风险分析和历史先例
- 明确提出强烈建议，但也清楚说明“最终由你决定”
- 记录决策，并提供演示脚本来支持用户后续推进

**你将学到：**
- 领导型代理如何框定战略决策
- 如何呈现带有权衡分析的选项
- 如何在建议中使用游戏开发先例和理论
- 如何记录决策（ADR）
- 如何把决策传递给受影响的部门

## 这些示例展示了什么

所有示例都遵循同一个**协作式工作流模式**：

```text
提问 -> 选项 -> 决策 -> 草稿 -> 批准
```

> **说明：** 这些示例以对话文本形式展示协作模式。  
> 在实际使用中，代理现在会在决策点使用 `AskUserQuestion` 工具，
> 以结构化选项选择器（带标签、描述和多选支持）的形式呈现决策。  
> 其模式是 **Explain -> Capture**：代理先在对话中解释分析，再通过结构化 UI 选择器收集用户决定。

### 展示出的协作行为

1. **代理会先问，再假设**
   - 设计代理会询问目标、约束和参考资料
   - 实现代理会澄清规格中的歧义
   - 领导型代理会先收集完整背景，再给出建议

2. **代理提供选项，而不是下命令**
   - 通常给出 2 到 4 个选项及其优缺点
   - 推理基于理论、先例和项目支柱
   - 可以给出推荐，但由用户作决定

3. **代理在定稿前先展示过程**
   - 设计草稿按章节展示
   - 架构方案在实现前展示
   - 战略分析在决策前展示

4. **代理在写文件前先获得批准**
   - 使用 `Write/Edit` 工具前，会明确询问“我可以写入 [file] 吗？”
   - 涉及多个文件时，会先列出所有受影响文件
   - 用户说“可以”之后才会创建文件

5. **代理会根据反馈迭代**
   - 用户修改会被立即吸收
   - 当用户调整推荐方案时，不会有防御性反应
   - 当用户改进了代理的建议时，会明确给予认可

## 如何使用这些示例

### 对新用户：
建议在第一次会话前先阅读这些示例。它们能帮助你建立对代理工作方式的现实预期：
- 代理是顾问，不是自主执行者
- 所有创意与战略决策都由你来做
- 代理提供的是专家级指导和可选方案

### 想理解特定工作流时：
- **想设计一个系统？** -> 阅读 `session-design-crafting-system.md`
- **想实现代码？** -> 阅读 `session-implement-combat-damage.md`
- **想做战略决策？** -> 阅读 `session-scope-crisis-decision.md`

### 用于培训时：
如果你在教别人如何使用这套系统，可以带着他们逐轮阅读一个示例，帮助他们理解：
- 什么样的问题才算好问题
- 如何评估代理给出的方案
- 什么时候该批准，什么时候该要求修改
- 如何在借助 AI 专业能力的同时保持创作主导权

## 所有示例中的共通模式

### 第 1-2 轮：先理解，再行动
- 代理先读取上下文（设计文档、规格、约束）
- 代理提出澄清问题
- 不做假设，也不靠猜测推进

### 第 3-5 轮：带着推理给出选项
- 给出 2 到 4 个不同方案
- 分析每个方案的优缺点
- 用理论或先例支撑分析
- 给出推荐，但把决策权交给用户

### 第 6-8 轮：迭代草稿
- 以增量方式展示工作成果
- 立即吸收反馈
- 主动标出边界情况或歧义

### 第 9-10 轮：批准与完成
- “我可以写入 [file] 吗？”
- 用户：“可以”
- 代理写入文件
- 代理提出下一步建议（测试、评审、集成）

## 自己试试看

看完这些示例后，可以做这个练习：

1. 选一个你的游戏系统（战斗、背包、成长等）
2. 让对应的代理来设计或实现它
3. 观察代理是否：
   - 会先提出澄清问题
   - 会给出带推理的选项
   - 会在定稿前先展示草稿
   - 会在写文件前请求批准

如果代理跳过了其中任何一步，你可以提醒它：

> "Please follow the collaborative protocol from docs/COLLABORATIVE-DESIGN-PRINCIPLE.md"
