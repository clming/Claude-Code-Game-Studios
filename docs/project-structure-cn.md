# Claude Code Game Studios 目录结构中文说明

本文档用中文解释这个仓库中各个主要目录和关键文件的作用，帮助你快速理解它不是一个具体游戏成品，而是一套面向游戏开发的 AI 协作模板。

## 一句话理解

这个仓库的目标是：

把一次普通的 Claude Code 会话，变成一个带有分工、流程、规范、检查机制的“虚拟游戏工作室”。

它提供的不是某一款游戏，而是一整套做游戏时可复用的 AI 工作流基础设施。

## 根目录主要内容

### `README.md`

项目总说明文档。

主要介绍：

- 这个项目为什么存在
- 它包含多少代理、技能、规则和钩子
- 整体工作方式和设计理念
- 如何开始使用

如果你第一次接触这个仓库，建议先读这个文件。

### `CLAUDE.md`

这是整个 AI 协作系统的主配置说明。

它定义了：

- 你用什么引擎和语言
- 项目结构参考
- 协作协议
- 编码标准
- 上下文管理方式

可以把它理解成“给 Claude Code 看的项目总章程”。

### `UPGRADING.md`

用于说明如果你已经在用旧版本模板，应该怎么升级到新版本。

### `LICENSE`

开源许可证，当前是 MIT。

### `production/`

生产管理相关目录，主要用来放：

- 冲刺计划
- 里程碑定义
- 发布记录
- 版本推进材料

它偏“项目管理”，不是游戏运行代码。

## `docs/` 目录

这个目录主要放面向人类阅读的说明文档、工作流说明和参考资料。

### `docs/WORKFLOW-GUIDE.md`

这是整套系统最重要的操作手册之一。

它按游戏开发阶段拆解了完整流程，例如：

- 从零开始构思
- 做概念设计
- 写系统设计文档
- 做原型验证
- 开发实现
- 测试与 QA
- 打磨与优化
- 本地化与无障碍
- 发布与上线
- 上线后热修复和复盘

如果你想知道“这个模板应该怎么跑起来”，它就是主路线图。

### `docs/COLLABORATIVE-DESIGN-PRINCIPLE.md`

这是协作原则文档。

它强调这个系统不是让 AI 自动替你做决定，而是：

- 先提问
- 再给方案
- 由你拍板
- 先看草稿
- 批准后再写入文件

也就是“用户驱动协作”，而不是“AI 自动作业”。

### `docs/examples/`

这里放的是示例对话和示例流程。

它的作用不是提供代码，而是让你看到：

- 一个设计类任务是怎么和 AI 配合完成的
- 一个实现类任务是怎么先澄清、再编码、再测试的
- 一个战略决策是怎么被分解、评估并落文档的

适合拿来学习这个系统的使用方式。

### `docs/engine-reference/`

这是引擎参考资料目录，非常关键。

因为大语言模型的知识有截止时间，而 Godot、Unity、Unreal 会持续更新，所以这里会保存“按版本固定”的引擎资料，防止 AI 给出过时建议。

它的作用是：

- 告诉代理当前项目锁定的引擎版本
- 列出破坏性变更
- 列出弃用 API
- 补充模型训练截止后新增的最佳实践
- 按模块拆分常见系统知识，比如 UI、渲染、物理、导航、音频等

#### `docs/engine-reference/godot/`

Godot 相关参考资料。

#### `docs/engine-reference/unity/`

Unity 相关参考资料。

#### `docs/engine-reference/unreal/`

Unreal Engine 相关参考资料。

如果你的项目决定使用某个引擎，对应目录就是 AI 参考的重点依据。

## `.claude/` 目录

这是整个仓库最核心的 AI 行为配置区。它决定 Claude Code 在这个项目里“怎么扮演一个游戏工作室”。

### `.claude/agents/`

这里是所有代理角色定义。

比如你会看到：

- `creative-director`
- `technical-director`
- `producer`
- `game-designer`
- `gameplay-programmer`
- `qa-lead`
- `unity-specialist`
- `unreal-specialist`

这些文件定义了每个代理的职责边界、关注重点和适用场景。

你可以把它理解成“岗位说明书”。

### `.claude/skills/`

这里是 slash commands，也就是你在 Claude Code 中输入 `/xxx` 时触发的技能。

例如：

- `/start`
- `/brainstorm`
- `/setup-engine`
- `/design-review`
- `/prototype`
- `/sprint-plan`
- `/release-checklist`

每个技能目录里通常有一个 `SKILL.md`，用于定义这个命令的执行流程。

你可以把它理解成“标准操作流程模板”。

### `.claude/rules/`

这里是路径级规则系统。

不同目录下的内容会被不同规则约束，例如：

- `gameplay-code.md`
- `ui-code.md`
- `network-code.md`
- `design-docs.md`
- `test-standards.md`

它的作用是让 AI 在不同类型文件里遵守不同工程规范，而不是一套规则打天下。

例如：

- 游戏逻辑要数据驱动
- UI 不应直接拥有核心游戏状态
- 网络代码要强调服务端权威
- 设计文档必须有完整章节

### `.claude/hooks/`

这里是自动化钩子脚本。

它们会在特定时机自动运行，例如：

- 会话开始
- 会话结束
- 提交前
- 推送前
- 资源文件改动后

它们主要用来做校验、提醒、记录和安全保护。

### `.claude/docs/`

这是给 Claude 和使用者参考的内部文档区。

里面包括：

- 代理清单
- 协作映射
- 快速开始说明
- 目录结构说明
- 规则说明
- 技术偏好说明
- 模板集合

如果说 `docs/` 更偏公开说明，那么 `.claude/docs/` 更偏系统内部运行资料。

### `.claude/settings.json`

这里通常是 Claude Code 的项目级配置中心，用来定义：

- hooks
- 权限策略
- 安全边界
- 一些自动行为

## 仓库中“将来会承载真实项目内容”的目录

虽然当前仓库更像模板，但它预设了真实游戏项目后续应该放内容的位置。

### `src/`

预留给游戏源代码。

通常会继续拆成：

- `src/core/`
- `src/gameplay/`
- `src/ai/`
- `src/networking/`
- `src/ui/`
- `src/tools/`

当前仓库里未必已经填满，但这是将来正式代码的主要位置。

### `assets/`

预留给游戏资源，例如：

- 美术资源
- 音频资源
- 特效
- Shader
- 数据文件

### `design/`

预留给游戏设计文档，例如：

- GDD
- 数值设计
- 世界观和叙事文档
- 关卡设计

### `tests/`

预留给测试代码和验证内容。

### `tools/`

预留给开发辅助工具、构建工具、流水线工具。

### `prototypes/`

预留给快速原型。

它和正式代码分开，是为了方便验证想法，而不污染主工程结构。

## 你可以怎么理解这个仓库

如果把它翻译成最直白的话：

- `docs/` 是说明书和参考书
- `.claude/agents/` 是虚拟团队成员
- `.claude/skills/` 是标准工作流命令
- `.claude/rules/` 是开发规范
- `.claude/hooks/` 是自动检查员
- `design/`、`src/`、`assets/`、`tests/`、`production/` 是你未来真正做游戏时要装内容的地方

所以它更像：

“一个可复制、可定制的 AI 游戏工作室操作系统”

而不是：

“某个已经写好的游戏项目”

## 推荐阅读顺序

如果你现在要开始理解这个项目，建议顺序如下：

1. 先读 `README.md`
2. 再读 `docs/WORKFLOW-GUIDE.md`
3. 再看 `docs/examples/README.md`
4. 最后结合 `.claude/skills/` 和 `.claude/agents/` 理解它怎么协作
