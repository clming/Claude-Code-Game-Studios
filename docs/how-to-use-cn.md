# Claude Code Game Studios 中文上手指南

本文档说明如何把这个仓库真正用起来，适合第一次接触这个模板的人。

## 先理解一件事

这个仓库不是“打开就能直接运行出一个游戏”的项目。

它更像一套工作台：

- 你决定要做什么游戏
- 你选择引擎
- 你和 AI 一起写设计文档、拆系统、做原型、写代码、做测试、做发布

也就是说，它是“做游戏的方法和协作框架”，不是“现成游戏成品”。

## 你第一次应该怎么开始

### 第 1 步：把它当成你自己新游戏项目的基础模板

通常你会这样使用它：

1. 克隆这个仓库
2. 把它作为你自己的游戏项目根目录
3. 在这个基础上继续增加你自己的 `src`、`assets`、`design`、`tests` 等内容

如果你只是阅读和学习，也可以先不改任何东西。

### 第 2 步：先看 3 份核心文档

建议先读：

1. `README.md`
2. `docs/WORKFLOW-GUIDE.md`
3. `docs/examples/README.md`

原因很简单：

- `README.md` 讲系统是什么
- `WORKFLOW-GUIDE.md` 讲从零到发布怎么走
- `examples` 讲真实对话里怎么配合 AI

### 第 3 步：在 Claude Code 中先执行 `/start`

这是最推荐的入口。

`/start` 的意义是让系统先判断你现在处于哪种阶段，例如：

- 还没有想法
- 只有模糊概念
- 已经有清晰设计
- 已经有现成项目

然后它会把你带到合适的下一步，而不是乱跳流程。

如果你什么都还没定，就从这里开始最稳。

### 第 4 步：尽快确定引擎

这个模板支持：

- Godot
- Unity
- Unreal Engine

一旦你决定引擎，就运行：

```text
/setup-engine
```

或者直接指定，例如：

```text
/setup-engine unity
```

它的作用是：

- 告诉系统你用什么引擎
- 让对应引擎代理成为主要参考
- 让 `docs/engine-reference/` 中对应版本资料发挥作用

如果引擎不明确，后面很多建议都会变得模糊。

## 两种常见起步路线

### 路线 A：你还没有游戏想法

建议按这个流程：

1. `/start`
2. `/brainstorm`
3. 让 AI 帮你整理概念文档
4. 定义游戏核心支柱
5. `/map-systems`
6. `/design-system`
7. `/design-review`
8. 开始原型或实现

这条路线适合从零开始。

### 路线 B：你已经知道要做什么游戏

建议按这个流程：

1. `/setup-engine`
2. 写或整理游戏概念文档
3. `/map-systems`
4. 按系统写 GDD
5. `/design-review`
6. `/sprint-plan`
7. 开始实现
8. `/code-review`

这条路线适合已经有方向的人。

## 一个实用的最小上手流程

如果你希望我把它压缩成最实际的“今天就能开始”的方式，可以按下面做。

### 场景：你想开始做自己的游戏

#### 1. 在 Claude Code 里先说清楚项目目标

你可以直接输入类似的话：

```text
我想做一个 2D 单机 roguelike，先帮我用这个模板建立最初项目结构。
```

#### 2. 然后执行：

```text
/start
```

#### 3. 确定引擎后执行：

```text
/setup-engine
```

#### 4. 让系统帮你形成概念文档

例如：

```text
请根据我们的想法，生成 game concept 和 game pillars 初稿。
```

#### 5. 拆系统

```text
/map-systems
```

这一步会把你的游戏拆成例如：

- 移动
- 战斗
- 背包
- UI
- 进度系统
- 敌人 AI

#### 6. 开始逐个设计系统

```text
/design-system combat-system
```

或者：

```text
/map-systems next
```

#### 7. 设计写完后做审查

```text
/design-review design/gdd/combat-system.md
```

#### 8. 再开始写代码

当设计文档比较完整后，再进入实现阶段。

例如你可以说：

```text
请根据 combat-system 的设计文档，开始实现基础伤害计算，并补充测试。
```

## 这个系统里最常用的命令

下面这些是你大概率最先会用到的：

### `/start`

新项目入口，帮助判断你当前处于哪个阶段。

### `/setup-engine`

配置游戏引擎和技术栈。

### `/brainstorm`

从零开始生成游戏概念、玩法方向、主题组合。

### `/map-systems`

把游戏拆成系统清单，并分析依赖关系和优先级。

### `/design-system`

围绕一个系统写设计文档。

### `/design-review`

检查设计文档是否完整、是否可实现、是否缺少边界条件。

### `/prototype`

做一个小型验证原型，用来测试玩法是否真的有趣、技术上是否可行。

### `/sprint-plan`

做开发冲刺计划。

### `/code-review`

做代码审查，找问题和风险。

### `/release-checklist`

准备发布前的检查清单。

## 正确使用这个系统的思路

### 1. 先设计，再实现

这个模板非常强调：

先把玩法和系统规则说清楚，再开始写代码。

这样做的好处是：

- 减少返工
- 防止 AI 乱猜需求
- 让设计、程序、测试可以对齐

### 2. 先做高风险原型，再投入正式开发

如果某个玩法是否好玩、某种技术是否能实现还不确定，就先用：

```text
/prototype
```

不要一开始就把所有东西写进正式代码。

### 3. 让不同角色做不同事情

这个仓库的优势就在于“角色分工”。

例如：

- 想玩法，用 `game-designer`
- 写实现，用 `gameplay-programmer`
- 看架构，用 `technical-director`
- 查性能，用 `performance-analyst`
- 做 QA，用 `qa-lead` 或 `qa-tester`

这样比把所有问题都丢给一个通用 AI 更稳定。

### 4. 把设计文档、代码、测试、发布都放进同一套流程

这个模板的价值不只是写代码，而是把：

- 想法
- 文档
- 代码
- 测试
- 生产计划
- 发布检查

放进同一套协作系统里。

## 如果你现在就要开始，我建议你这样做

这是我给你的最实用起步顺序：

1. 先读 `README.md`
2. 再读 `docs/project-structure-cn.md`
3. 然后在 Claude Code 中运行 `/start`
4. 立刻确定引擎并运行 `/setup-engine`
5. 让系统帮你生成概念文档和 game pillars
6. 用 `/map-systems` 拆系统
7. 先设计一个核心系统，例如战斗或移动
8. 用 `/design-review` 检查设计
9. 再进入实现

## 如果你是 Unity 用户，特别建议

因为你现在打开的是 Unity 相关参考文档，所以如果你准备走 Unity 路线，建议你优先做这几件事：

1. 把 `CLAUDE.md` 里的引擎信息改成 Unity
2. 检查 `docs/engine-reference/unity/` 下的版本说明
3. 优先使用 Unity 专项代理和相关技能
4. 在正式写代码前，先确认 UI、渲染、Addressables 这些模块参考文档和你的项目版本一致

## 你下一步最值得做的事

如果你想真正开工，而不是只看说明，我建议下一步直接做下面这件事：

让 AI 帮你把当前仓库初始化成“你的游戏项目版本”。

你可以直接对我说：

```text
请根据这个模板，帮我初始化一个 Unity 2D 游戏项目的最小目录和初始文档。
```

或者：

```text
请把这个模板改造成适合我当前项目的工作台，我的项目是一个 Unity 单机动作游戏。
```

这样我们就可以从“理解模板”进入“真正开始搭项目”。
